The Google API centers on **Resources** — objects you create, read, update, and delete through the API. Every resource shares a common set of fields that appear consistently across the platform, so once you understand the shape of one resource, you'll recognize the pattern everywhere. This shared structure makes it easier to write generic client code, log data reliably, and reason about state across your integration.

## The Resource object

When you create or retrieve a resource, the API returns a JSON object with the following shape:

```json
{
  "id": "res_01J9KXPQ3N2M4R5T6W7Y8Z",
  "name": "My Resource",
  "description": "An optional description",
  "status": "active",
  "metadata": {},
  "created_at": "2024-01-15T10:30:00Z",
  "updated_at": "2024-01-15T11:00:00Z"
}
```

Resource-specific fields — those unique to a particular resource type — are returned alongside these common fields in the same top-level object.

## Field conventions

The following fields appear on every resource returned by the Google API:

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique identifier with a resource-type prefix (e.g. `res_`) |
| `created_at` | string (ISO 8601) | When the resource was created |
| `updated_at` | string (ISO 8601) | When the resource was last modified |
| `status` | string | Resource lifecycle state: `active`, `inactive`, or `deleted` |
| `metadata` | object | Up to 20 key-value pairs for your own use |

## IDs

Every resource has a globally unique `id` generated by the API at creation time. IDs are prefixed with a short string that identifies the resource type — for example, `res_` for generic resources. This prefix convention means you can tell at a glance what kind of object an ID refers to, which is especially useful when you store IDs from multiple resource types in the same data store. IDs are immutable: once assigned, they never change and are never reused.

## Metadata

The `metadata` field lets you attach up to 20 arbitrary key-value pairs to any resource. Use it to store internal references, tags, or any contextual data that matters to your application — for example, a customer ID from your own database or a deployment environment label. Keys and values must both be strings.

To add or update metadata, send a `PATCH` request with the fields you want to set. Existing keys you omit are preserved; set a key's value to `null` to remove it.

```bash
curl -X PATCH https://api.google.com/v1/resources/res_01J9KXPQ3N2M4R5T6W7Y8Z \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "metadata": {
      "customer_id": "cust_789",
      "environment": "production"
    }
  }'
```

## Timestamps

All timestamps in the Google API use **ISO 8601** format and are expressed in **UTC**. For example: `2024-01-15T10:30:00Z`. The trailing `Z` confirms the value is UTC — there are no timezone offsets to normalize. When you display timestamps to end users, convert from UTC to their local timezone in your application layer. Both `created_at` and `updated_at` are set by the server and are read-only; you cannot supply them in create or update requests.
