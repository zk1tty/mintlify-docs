The Google API is a REST service built around predictable, resource-oriented URLs. Every piece of data you work with — whether you're creating a new resource, fetching an existing one, or deleting it — lives at a stable URL that follows the same consistent pattern. You interact with the API by sending standard HTTP requests and receiving structured JSON responses, making it straightforward to integrate with any language or HTTP client.

## RESTful design

The Google API models everything as a **resource** — a discrete object with a unique ID and a well-defined URL. You read, create, modify, and delete resources using the standard HTTP methods your tooling already understands. Because the API is stateless, each request carries all the information needed to fulfill it; there are no sessions or server-side state to manage. This predictability means you can reason about any endpoint in the API using the same mental model.

## Base URL

All API requests target the following base URL:

```text
https://api.google.com/v1
```

Every endpoint path is appended to this base. For example, to work with resources, you'll call `https://api.google.com/v1/resources`.

## Request format

All request bodies must be valid JSON. Set the `Content-Type` header to `application/json` on every request that includes a body. Authenticate by passing your API key as a Bearer token in the `Authorization` header.

Here is a template you can adapt for any endpoint:

```bash
curl -X POST https://api.google.com/v1/resources \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "My Resource",
    "description": "An optional description"
  }'
```

## Response format

Every response from the Google API is JSON. Successful responses wrap the returned object in a `data` field and include a `meta` object containing a unique `request_id` you can use when troubleshooting with support.

```json
{
  "data": { "..." },
  "meta": { "request_id": "req_abc123" }
}
```

Error responses follow a separate, consistent structure — see the [Error Handling](/guides/error-handling) guide for details.

## HTTP methods

The Google API uses standard HTTP methods to convey the intent of each operation:

| Method | Use | Example |
|--------|-----|---------|
| `GET` | List resources | `GET /resources` |
| `POST` | Create a resource or register a webhook | `POST /resources`, `POST /webhooks` |
| `GET` | Retrieve a single resource | `GET /resources/{id}` |
| `PATCH` | Update a resource | `PATCH /resources/{id}` |
| `DELETE` | Delete a resource | `DELETE /resources/{id}` |

`PATCH` requests are partial — you only need to include the fields you want to change. Fields you omit remain unchanged.
